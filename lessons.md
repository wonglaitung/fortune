## 🔴 核心警告

### 高准确率通常是数据泄漏的信号
| 等级 | 准确率范围 | 判断 |
|------|-----------|------|
| 随机/平衡基线 | ≈50% | 正常 |
| 常见弱信号 | ≈51–55% | 正常 |
| 有意义的改进 | ≈55–60% | 良好 |
| 非常好/罕见 | ≈60–65% | 优秀 |
| **异常高（需怀疑）** | **>65%** | **可能数据泄漏** |

### 模型可信度评估
| 模型 | 可信度 | 说明 |
|------|--------|------|
| **CatBoost 20天** | ⭐⭐⭐⭐⭐ | 高可信度，**推荐使用** |
| **CatBoost 5天** | ⭐⭐⭐ | 中等可信度，谨慎使用 |
| **CatBoost 1天** | ⭐ | 低可信度，**不推荐**（严重过拟合） |
| **LSTM/Transformer** | ⭐ | 低可信度，**不推荐**（F1≈0） |
| **融合模型** | ⭐ | 低可信度，**不推荐**（信号稀释） |

### Walk-forward验证是唯一可信的评估方法
- ❌ 简单评估：使用训练数据评估，结果虚高（83-90%）
- ✅ Walk-forward：每个fold重新训练，严格时序分割，反映真实预测能力

## 📊 关键指标定义

| 指标 | 定义 | 使用场景 |
|------|------|----------|
| **胜率** | 盈利交易数 / 买入信号总数 | 评估买入信号质量 |
| **准确率** | 预测正确数 / 总交易机会数 | 评估方向预测能力 |
| **正确决策比例** | 正确决策数 / 总决策数 | 综合评估决策质量 |

**重要**：高准确率 ≠ 高收益。准确率86.21%，买入胜率23.04%，平均收益率-1.94%（避雷模式）

## 🎯 交易信号与异常关联性验证经验 ⭐ 新增（2026-04-03）

### 验证方法论
| 方法 | 用途 | 业界标准 | 实现状态 |
|------|------|----------|----------|
| 相关性分析（Pearson/Spearman） | 评估异常与信号的相关程度 | ✅ 标准方法 | ✅ 已实现 |
| Granger因果检验 | 验证异常是否预示信号变化 | ✅ Two Sigma/Binance | ✅ 已实现 |
| 时间序列交叉相关分析 | 分析异常前置/滞后效应 | ✅ TradingView | ✅ 已实现 |
| 事件研究法 | 分析异常前后信号表现 | ✅ 量化研究 | ✅ 已实现 |
| 回测对比 | 验证异常期间信号质量 | ✅ 量化交易 | ✅ 已实现 |

### 关键发现
1. **异常与信号显著相关**：
   - 相关系数0.1843（p < 0.05）
   - 统计显著，非随机产生
   - 符合业界预期（0.15-0.25）

2. **异常是信号的Granger原因** ⭐⭐⭐：
   - 整体显著（4个滞后期显著）
   - 最强滞后期：lag_4（p=0.007）
   - 异常可以预测信号变化（1-5天窗口）

3. **异常后信号质量提升** ⚠️ 需要修正：
   - 收益率提升+245%（-0.31% → 0.45%）
   - 但波动率提升+79%（3.77% → 6.76%）
   - **风险调整后夏普比率仍为负值**（-1.19 → -0.87）
   - **结论：异常期间风险增加，需要更谨慎**

4. **异常是"放大器"而非"方向指示器"** ⭐⭐⭐：
   - 异常后波动率大幅增加（+79%）
   - 异常后胜率接近随机（52.38%）
   - **不是"异常后会涨"，而是"异常后波动加大"**
   - **结论：异常期间应该提高阈值、降低仓位、收紧止损**

| 无异常 | 0.60 | 0.60 | 1.0x | 正常市场环境 |

### 风险与局限性
1. **样本量有限**：
   - 仅10个异常事件
   - 建议：每季度重新验证

2. **单一资产**：
   - 仅验证ETH-USD
   - 建议：扩展到BTC、黄金、港股

3. **时间跨度不足**：
   - 仅2年数据（2024-2025）
   - 建议：扩展到5-10年

### 最佳实践
1. **多维度验证**：相关性、因果性、事件研究、回测对比
2. **统计显著性检验**：p值 < 0.05，拒绝零假设
3. **风险调整收益**：优先考虑夏普比率，而非绝对收益率
4. **动态策略调整**：根据异常严重程度调整阈值、仓位、止损
5. **持续监控**：每季度重新验证，及时调整策略
6. **风险管理优先**：高波动+高不确定性=高风险，必须降低仓位

   - Granger因果检验证明（4个滞后期显著）

   - 不能只看收益率，必须看波动率（标准差）
   - 不能只看胜率，必须看夏普比率（风险调整收益）

## ⚠️ 常见错误

### 数据泄漏检查
**高风险特征**（必须使用.shift(1)）：
- BB_Position、Price_Percentile、Intraday_Amplitude
- Price_Ratio_MA5/20/50、Price_Pct_20d
- Days_Since_High/Low_20d、Intraday_Range
- Support_120d、Resistance_120d
- 所有 `.rolling()` 计算的特征（ATR、成交量、波动率等）

**修复效果**：
| 模型 | 修复前准确率 | 修复后准确率 | 标准差变化 |
|------|------------|------------|----------|
| CatBoost 20天 | 68.33% | **60.99%** | 6.45% → **2.00%** (-69%) ✅ |
| LightGBM 20天 | 58.93% | **58.33%** | 8.35% → **7.64%** (-8.5%) ✅ |
| GBDT 20天 | 57.94% | **56.10%** | 8.22% → **4.93%** (-40%) ✅ |

**经验**：
1. 准确率下降是正常的，反映真实预测能力
2. 标准差降低表明模型更稳定，长期表现更可靠
3. 修复一个模型的特征工程代码，所有使用该代码的模型都受益

### 夏普比率计算
```python
# 错误：分子分母不匹配
annualized_return = avg_return * (252 / horizon)  # avg_return包含所有交易
return_std = buy_signals['strategy_return'].std()  # 只用买入信号

# 正确：统一使用买入信号数据
avg_return_for_sharpe = buy_signals['strategy_return'].mean()
annualized_return = avg_return_for_sharpe * (252 / horizon)
return_std = buy_signals['strategy_return'].std()
sharpe = (annualized_return - 0.02) / annualized_std
```

**修复原因**：不买入交易的0收益会稀释平均收益率，导致夏普比率不准确（通常偏低）

### 回撤计算（多周期持有）
```python
# 错误：使用重叠样本（导致极端回撤值）
drawdown = (1 + df['strategy_return']).cumprod() - 1

# 正确：horizon>1时使用非重叠样本
if horizon > 1:
    non_overlapping = df.iloc[::horizon].copy()
    drawdown = (1 + non_overlapping['strategy_return']).cumprod() - 1
```

**验证结果**（银行股板块）：
- 修正前平均回撤：-65.45%（不合理）
- 修正后平均回撤：-13.12%（符合银行股特性）

### 验证方法
- 12个Fold，12个月训练窗口，1个月测试窗口
- 置信度阈值：0.65
- 验证期间：2024-01-01 至 2025-12-31

### 核心发现

1. **消费股是唯一符合条件的板块**：
   - 夏普比率 > 0：0.7445
   - 买入胜率 > 50%：54.80%
   - 稳定性高：收益率标准差仅1.22%

   - 低波动板块（消费、银行）：胜率较高（接近或超过50%）

4. **稳定性vs收益权衡**：
   - 消费股：稳定性高，收益率低（5.10%）
   - 银行股：收益率高（45.92%），稳定性中等

**为什么全量特征更好**：

- 保留所有特征，避免信息丢失（392个特征）

**最终推荐**：
- ✅ **使用全量特征（892个特征）** ⭐⭐⭐⭐⭐
- 训练时省略`--use-feature-selection`参数
- 唯一代价：训练速度较慢（每个Fold多约1分钟）

## 🔧 模型可重现性优化

### 问题背景
两台机器在不同时间训练模型，预测结果存在显著差异：

**差异情况**：
- 4只股票预测方向发生变化（1288.HK、0981.HK、1088.HK、9988.HK）
- 预测概率差异最大达0.1619（1288.HK 农业银行：0.6591 → 0.4972）

**根本原因**：
1. **CatBoost的随机性**：行采样（75%）和列采样（70%）导致每次训练使用不同的样本和特征子集
2. **临界区域敏感性**：预测概率在0.48-0.55之间的股票最容易受训练数据微小差异的影响
3. **训练数据时间差异**：不同时间获取的收盘价数据可能存在微小差异

### 解决方案

为所有模型的 `train()` 方法添加固定随机种子：

```python
def train(self, codes, start_date=None, end_date=None, horizon=1, use_feature_selection=False):
    # 设置固定随机种子（确保模型训练的可重现性）
    np.random.seed(42)
    random.seed(42)
    
    # 继续训练逻辑
    ...
```

### 业界最佳实践

**研究/开发阶段**：
- ✅ 使用固定随机种子（可重现性）
- 好处：便于调试和对比

**验证阶段**：
- ✅ 多次训练，评估稳定性
- 好处：发现过拟合问题

**生产部署**：
- ✅ 固定种子（保守）⭐ 推荐
- 备选：集成模型（稳定性+泛化能力兼顾，但训练时间增加3倍）

### 经验教训

1. **随机性影响显著**：CatBoost的行采样和列采样会导致模型差异
2. **固定种子必要性**：生产环境必须使用固定随机种子，这不是过度设计，而是业界标准
3. **临界区域敏感性**：预测概率在0.48-0.55之间的股票最容易变化
4. **符合CatBoost官方建议**："For production deployment, we recommend using a fixed random seed"

## 📝 工作流优化经验（2026-04-03）

### 功能重复检查的重要性
**问题**：多个工作流执行相同脚本导致资源浪费和维护困难

**案例**：
- `ethereum-anomaly-detection.yml` 和 `hourly-crypto-monitor.yml` 都执行 `crypto_email.py`
- 执行频率重叠（每小时+凌晨2点）
- 依赖重复，难以维护

**解决方案**：
1. 合并工作流到一个文件
2. 通过调度时间和参数区分执行模式
3. 使用 `if` 条件判断执行不同的步骤

**最佳实践**：
```yaml
- name: Quick mode (hourly)
  if: github.event.schedule != '0 2 * * *'
  run: python3 script.py --mode quick

- name: Deep mode (2 AM)
  if: github.event.schedule == '0 2 * * *'
  run: python3 script.py --mode deep
```

### 参数传递优化：命令行参数 vs 环境变量

**环境变量的缺点**：
1. 依赖 `set_key.sh` 配置，用户需要手动设置
2. 不直观，难以在命令行看到传递的参数
3. 测试时需要手动导出变量

**命令行参数的优势**：
1. 更灵活，可以手动指定任意值
2. 更清晰，参数直接在命令行可见
3. 更标准，符合 Unix 工具最佳实践
4. 更易于测试和调试

**迁移示例**：
```python
# 旧方法（环境变量）
detection_mode = os.environ.get('ANOMALY_DETECTION_MODE', 'quick')
use_deep_analysis = (detection_mode == 'deep')

# 新方法（命令行参数）
import argparse
parser = argparse.ArgumentParser()
parser.add_argument('--mode', choices=['quick', 'deep'], default='quick')
args = parser.parse_args()
use_deep_analysis = (args.mode == 'deep')
```

**结论**：对于脚本运行模式、配置选项等，优先使用命令行参数而非环境变量

### 异常检测分层策略

**双层检测架构**：
| 层级 | 频率 | 方法 | 目的 | 性能 |
|------|------|------|------|------|
| Layer 1 | 每小时 | Z-Score | 实时检测价格/成交量异常 | 快速 |
| Layer 2 | 凌晨2点 | Isolation Forest | 深度分析7天数据，发现复杂异常 | 全面 |

**关键要点**：
1. 基础检测（Z-Score）始终执行，覆盖范围广
2. 深度检测（Isolation Forest）仅在某些时机执行，避免计算成本
3. 两层检测结果整合，确保异常不会遗漏
4. 动态邮件标题反映实际检测内容

**实现方式**：
```python
def run_anomaly_detection(prices, use_deep_analysis=False):
    # 始终执行 Z-Score
    zscore_detector = ZScoreDetector()
    zscore_anomalies = zscore_detector.detect()
    
    # 仅在深度模式执行 Isolation Forest
    if use_deep_analysis:
        if_detector = IsolationForestDetector()
        if_anomalies = if_detector.detect()
```

### 邮件发送条件优化

**问题**：只检查交易信号，忽略异常检测结果

**解决方案**：同时检查交易信号和异常
```python
has_important_content = has_signals

# 检查异常（只发送 high/medium 级别）
if anomaly_result and anomaly_result['has_anomaly']:
    severity = anomaly_result.get('severity', 'low')
    if severity in ['high', 'medium']:
        has_important_content = True
```

**动态邮件标题**：
```python
if has_signals and anomaly_result and anomaly_result['has_anomaly']:
    subject = "加密货币更新 - 交易信号 + 异常检测"
elif has_signals:
    subject = "加密货币更新 - 交易信号提醒"
elif anomaly_result and anomaly_result['has_anomaly']:
    subject = f"加密货币更新 - 异常检测提醒 [{severity_emoji} {severity.upper()}]"
```

**关键要点**：
1. 邮件标题应准确反映内容类型
2. 过滤低优先级异常（low级别），避免垃圾邮件
3. 优先发送重要信息（交易信号、高/中级别异常）
