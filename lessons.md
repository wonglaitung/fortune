### 0. Volume/Turnover 数据泄漏 - IC/Rank IC 分离的根因 ⭐⭐⭐（更新 2026-05-06）

**问题**：15+ 个成交量相关特征存在数据泄漏，导致 IC 为正但 Rank IC 为负

**背景**：
- 成交量（Volume）与价格波动高度相关，当日成交量是当日涨跌的"事后信号"
- 模型通过当日成交量"偷看"了当日价格走势，制造了 IC 虚高的假象
- 实盘预测时无法获得当日收盘成交量，排序逻辑完全失效

**受影响的特征**（已修复）：

| 特征 | 问题 | 修复 |
|------|------|------|
| `VWAP` | `Volume` 未 shift | `df['Volume'].shift(1)` |
| `CMF` | `Volume` 和 `Close` 未 shift | 全部使用 shift(1) |
| `OBV` | 使用当日 `Volume` 推断当日涨跌 | 整体 shift(1) |
| `Vol_Ratio` | `Volume` 未 shift | `df['Volume'].shift(1)` |
| `Vol_Z_Score` | `Volume` 未 shift | `df['Volume'].shift(1)` |
| `Turnover` | `Close * Volume` 含当日数据 | 两者都 shift(1) |
| `Turnover_Z_Score` | `Turnover` 未 shift | 已修复 |
| `Volume_Ratio_5d/20d` | `Volume` 未 shift | `df['Volume'].shift(1)` |
| `Volume_Ratio_7d/120d` | `Volume` 未 shift | `df['Volume'].shift(1)` |
| `Price_*_Volume_*` | `Turnover.pct_change()` 含当日数据 | 使用滞后数据 |

**为什么 IC 正但 Rank IC 负？**

1. **IC（Pearson 相关性）**：衡量预测值与收益的线性关系
   - 模型通过当日成交量"锚定"了当日涨跌
   - 回测时 IC 看起来很好（虚高）

2. **Rank IC（Spearman 相关性）**：衡量预测排名与收益排名的关系
   - 实盘时无法获得当日收盘成交量
   - 模型学到的"成交量-价格"逻辑完全错位
   - 排序变成随机乱炖，甚至反向

**核心教训**：
- **成交量是价格的事后信号**，必须严格使用 shift(1)
- **OBV 是累加指标**，必须整体滞后，否则模型可 100% 推断当日涨跌
- **IC 虚高是数据泄漏的典型症状**，需检查 Rank IC 是否一致
- **P9 特征剔除失败的原因**：模型把泄露特征当"救命稻草"，剔除宏观特征后变本加厉挖掘噪声

**代码位置**：

- `ml_services/ml_trading_model.py:822-837`（量价关系特征）

### 1. CatBoostRanker 排序模型的概率变换 ⭐（更新 2026-05-04）

**问题**：CatBoostRanker 的原始分数分布集中，sigmoid 变换后概率过于集中在 0.5 附近

**背景**：
- CatBoostRanker 使用 YetiRank/YetiRankPairwise 损失函数，输出原始排序分数
- 原始分数范围通常很小（例如 [-0.15, 0.22]，标准差 ~0.05）
- 直接 sigmoid 变换后，概率集中在 [0.46, 0.55] 附近
- 置信度阈值 0.55 会过滤掉几乎所有信号

**解决方案**：

1. **温度参数归一化**
   ```python
   # 使用分数标准差作为温度，归一化后 sigmoid 输出范围更广
   temperature = score_std  # 而非 1.0 / score_std
   scaled_scores = scores / temperature
   proba_col1 = expit(scaled_scores)
   ```

2. **分类特征处理**
   ```python
   # predict_proba 中必须正确传递 cat_features
   categorical_features = [self.feature_columns.index(col) for col in self.categorical_encoders.keys()]
   test_pool = Pool(data=test_df.values, cat_features=categorical_features if categorical_features else None)
   ```

**五版本验证对比**：

| 指标 | CatBoostClassifier | YetiRank | YetiRankPairwise | P5 软标签 | P7 特征优化 | 目标 |
|------|---------------------|----------|------------------|-----------|-------------|------|
| IC | -0.0216 | +0.0122 | **+0.0066** | -0.0022 | +0.0033 | >0.01 |
| Rank IC | -0.0012 | -0.0060 | **+0.0038** | -0.0063 | -0.0025 | >0.02 |
| 准确率 | 59.38% | 49.69% | 49.80% | 48.57% | 49.29% | - |
| 夏普比率 | 0.8291 | 0.7945 | **0.8103** | 0.7146 | 0.7786 | >0.8 |
| 索提诺比率 | 3.54 | 5.61 | 5.06 | 3.14 | **5.2051** | >5.0 |

**失败尝试**：
- ❌ **MinMax 归一化**：将分数直接映射到 [0,1]
  - 结果：IC 从 +0.0066 降至 +0.0039，Rank IC 从 +0.0038 降至 +0.0014
  - 原因：MinMax 丢失了分数的相对距离信息，sigmoid 保留了非线性关系
- ❌ **软标签（截面排名百分位）**：将标签从原始收益率改为排名百分位（0~1）
  - 结果：IC 从 +0.0066 降至 -0.0022，Rank IC 从 +0.0038 降至 -0.0063
  - 原因：软标签丢失收益幅度信息，YetiRankPairwise 无法区分"大幅跑赢"和"小幅跑赢"
- ⚠️ **截面特征扩展 + 宏观交叉特征剔除**（P7）
  - 结果：IC 从 +0.0066 降至 +0.0033，Rank IC 从 +0.0038 降至 -0.0025
  - 原因：截面特征扩展引入噪声，宏观交叉特征剔除过度导致模型容量不足

**核心教训**：
- **排序模型优化排序而非分类准确率**，准确率下降是预期的
- **YetiRankPairwise 优于 YetiRank**：成对比较式排序更适合截面选股场景
- **温度参数应使用 score_std**，而非 1.0/score_std（后者会使概率更集中）
- **分类特征必须在 predict_proba 中正确处理**，否则会报错
- **MinMax 归一化不适合 Ranker 分数转概率**，自动温度 sigmoid 是最佳策略
- **软标签不适合 YetiRankPairwise**：丢失幅度信息后，成对比较无法区分强弱程度
- **索提诺比率提升**表明风险调整收益更好
- **截面特征非越多越好**：质量比数量重要，低质量特征会稀释有效信号
- **宏观交叉特征有双重作用**：既含宏观成分（干扰），也含相对强度信息（有用）
- **P4（YetiRankPairwise + 原始收益率）仍是当前最优配置**

**代码位置**：
- `ml_services/ml_trading_model.py:7481-7510`（CatBoostRankerModel.predict_proba）
- `ml_services/ml_trading_model.py:7198-7308`（use_soft_label 参数）

### 1. 所有模型必须使用一致的特征工程逻辑 ⭐（更新 2026-05-03）

**问题**：不同模型使用不同的特征计算逻辑，导致预测结果不可比

**背景**：
- 系统包含三种模型：CatBoost、LightGBM、GBDT
- CatBoost 已实现截面特征（`_CS_Pct`, `_CS_ZScore`）
- LightGBM/GBDT 此前仅支持逐只预测，截面特征退化为 0.5/0.0
- 这导致相同股票、相同日期的预测结果因模型而异

**不一致的后果**：

| 场景 | CatBoost | LightGBM/GBDT | 问题 |
|------|----------|---------------|------|
| 单股预测截面特征 | 0.5/0.0 回退 | 0.5/0.0 回退 | ✅ 一致 |
| 批量预测截面特征 | 联合计算 | 逐只计算 | ❌ 不一致 |
| 截面特征值 | 真实排名 | 常数 0.5 | ❌ 分布差异 |
| 模型融合结果 | 不可靠 | 不可靠 | ❌ 信号冲突 |

**解决方案**：

1. **统一批量预测架构**
   ```python
   def predict_batch(self, codes, predict_date=None, horizon=None):
       # 阶段1：逐只提取原始特征
       all_features = {}
       for code in codes:
           stock_df = self._extract_raw_features_single(code, ...)
           all_features[code] = stock_df
       
       # 阶段2：合并后联合计算截面特征
       combined = pd.concat(all_features.values())
       combined = self._calculate_cross_sectional_features(combined)
       
       # 阶段3：逐只预测
       for code in all_features.keys():
           stock_data = combined[combined['Code'] == code]
           result = self._predict_from_features(code, stock_data)
   ```

2. **统一截面特征列表**
   - 所有模型使用相同的 `CROSS_SECTIONAL_PERCENTILE_FEATURES`（55个）
   - 所有模型使用相同的 `CROSS_SECTIONAL_ZSCORE_FEATURES`（43个）
   - 确保特征名称、计算方式完全一致

3. **统一回退机制**
   - 训练时保存 `cs_feature_stats`（截面特征统计量）
   - 单股预测时使用训练集均值填充缺失值
   - 最终回退：`_CS_Pct` 用 0.5，`_CS_ZScore` 用 0.0

4. **统一模型持久化**
   ```python
   def save_model(self, filepath):
       model_data = {
           # ... 原有字段
           'use_cross_sectional_percentile': self.use_cross_sectional_percentile,
           'use_cross_sectional_zscore': self.use_cross_sectional_zscore,
           'cs_feature_stats': self.cs_feature_stats,  # 关键：持久化统计量
       }
   ```

**核心教训**：
- **多模型系统必须保持特征工程逻辑一致**，否则预测结果不可比
- 截面特征必须在所有股票数据上联合计算，单只股票无法计算
- 批量预测架构是截面特征正确性的前提
- 单股预测必须有合理的回退机制（训练集统计量 > 默认值）
- 模型持久化必须包含截面特征统计量，否则加载后回退失效

**代码位置**：
- `ml_services/ml_trading_model.py`（LightGBMModel、GBDTModel、CatBoostModel）

### 1. 网络特征数据泄漏防护 ⭐（新增 2026-05-02）

**问题**：网络特征计算存在数据泄漏风险

**泄漏点**：
- `build_returns_dataframe()` 直接使用当日收益率数据
- `calculate_node_deviation()` 包含当日收益率
- 如果用 T 日数据构建网络预测 T 日收益，存在泄漏

**修复措施**：

| 修复项 | 修复前 | 修复后 |
|--------|--------|--------|
| 收益率 DataFrame | `df['Return']` | `df['Return'].shift(1)` |
| 节点偏离度 | `returns_df.iloc[-window:]` | `returns_df.iloc[-window-1:-1]` |
| 参数统一 | 无 | `for_prediction=True` |

**验证方法**：
- 所有网络特征计算调用 `build_returns_dataframe(stock_data, for_prediction=True)`
- 确保不使用 `returns_df.iloc[-1]`（当日数据）

**教训**：
- **网络特征作为独立风险监控工具，不直接放入个股预测模型**
- 即使用于风险监控，也需确保时间偏移，避免在实盘中误用
- 波动率网络密度使用 `rolling().std()`，基于历史数据，本身无泄漏风险
- 网络结构变化缓慢（<0.1%/日），使用 T-1 日数据不影响分析结果

### 2. 数据泄漏阈值因模型而异 ⭐（更新 2026-04-29）

**问题**：恒指和个股的数据泄漏阈值不同

**原因**：
- 恒指是"均值"，噪声被对冲，预测更容易
- 个股受公司特定事件影响，预测难度更高
- 因此恒指的"正常准确率上限"高于个股

**阈值标准**：

| 模型类型 | Walk-forward 准确率正常范围 | 数据泄漏信号 |
|---------|---------------------------|-------------|
| 个股 | 50-65% | >65% |
| 恒指 | 60-82% | >80%（增强模型） |

**最新验证结果（2026-04-29，特征残差化）**：
- 个股准确率：60.77%（✅ 正常，<65%）
- 最高 fold 准确率：65.93%（✅ 正常）

**教训**：
- 不要用同一标准衡量不同类型的模型
- 恒指 80% 准确率是正常的，个股 65% 就要警惕

### 2. IC 为负值表示选股能力有限 ⭐（新增 2026-04-29，更新 2026-05-02）

**问题**：实施特征残差化后，IC 为负值（-0.0181）

**验证数据**：

| 指标 | 数值 | 说明 |
|------|------|------|
| IC | -0.0181 | 预测概率与实际收益弱负相关 |
| Rank IC | -0.0124 | 预测排名与实际收益排名弱负相关 |
| 预测分散度 | 0.2393 | ✅ 有区分度，避免"全涨全跌" |

**原因分析**：
- IC 衡量的是"选股能力"，即预测概率高的股票是否实际收益也高
- 负值表示预测概率高的股票反而收益略低
- 这与 lessons.md 中"个股预测信号不可靠"的结论一致

**深层原因：Regime Shift 导致特征方向翻转**：

| 问题 | 说明 |
|------|------|
| **特征方向不稳定** | 2020年"波动率↑ → 下跌"，2024年可能变成"波动率↑ → 上涨"（逼空行情） |
| **旧数据干扰** | 2020-2022 年的数据权重过高，干扰 2024-2025 年的预测 |
| **绝对量级失效** | 波动率=0.03 在2020年是"高波动"，在2024年可能是"低波动" |

**解决方案（2026-05-02 实施，消融实验验证）**：

| 方案 | 单独效果 | 说明 | 推荐设置 |
|------|---------|------|---------|
| **单调约束** | IC +0.0062 (↑32%) | 强制特征方向不变 | ✅ **推荐开启** |
| **时间衰减权重** | IC -0.0005 (基本不变) | 降低旧数据权重 | ✅ 保留 λ=0.5 |
| **滚动百分位** | IC -0.0025 (↓13%) | 绝对值转历史百分位 | ❌ **已关闭** |

**消融实验结论（2026-05-02）**：

| 实验 | IC | 说明 |
|------|-----|------|
| 基线（无修复） | 0.0191 | 基准 |
| 仅单调约束 | 0.0253 | ✅ 最佳，IC 提升 32% |
| 仅时间衰减 | 0.0186 | ➖ 基本不变 |
| 仅滚动百分位 | 0.0166 | ⚠️ IC 下降 13% |
| 全部修复 | -0.0201 | ❌ 组合后 IC 变负值 |

**关键发现**：
- **滚动百分位不适合个股预测模型**：时间序列自归一化丢失绝对量级信息
- **单调约束是业界最佳实践**：Man-AHL、Two Sigma 等机构都使用
- **组合效应负面**：滚动百分位改变了单调约束的特征分布，约束失效
- **波动率约束在相对标签模型下不稳定**：牛市+1，熊市-1，应移除

**单调约束优化（2026-05-03 基于业界实践）**：

| 约束类型 | 数量 | 理由 |
|----------|------|------|
| **保留** | 5个 | RS_Ratio(+1), RSI(-1), MACD_histogram(+1) |
| **移除** | 25个 | 波动率、股息、情感（方向不稳定或反向） |

**代码位置**：`ml_services/ml_trading_model.py:4084-4108`

**截面百分位方案（已实施，2026-05-03）**：

如果需要百分位化，应使用**截面百分位**而非滚动百分位：

| 特性 | 滚动百分位（已关闭） | 截面百分位（已启用） |
|------|---------------------|----------------------|
| **计算方式** | `df[feat].rolling(252).rank(pct=True)` | `df.groupby('Date')[feat].rank(pct=True)` |
| **比较对象** | 该股票过去252天的值 | 当日所有股票的值 |
| **信息保留** | ❌ 丢失绝对量级 | ✅ 保留相对排名 |
| **适用场景** | 时间序列预测 | 截面选股 |
| **与相对标签匹配** | ❌ 不匹配 | ✅ 完美匹配 |
| **参数** | `use_rolling_percentile=False` | `use_cross_sectional_percentile=True` |

**截面百分位消融实验（2026-05-03 验证）**：

| 指标 | 基线（无截面百分位） | 截面百分位 | 变化 | 评估 |

| **索提诺比率** | - | **5.99** | - | ✅ 优秀 |

**消融结论**：
- ✅ **夏普比率显著提升**：风险调整收益改善 11.6%
- ✅ **最大回撤改善**：风险控制更好
- ⚠️ **IC 未改善**：选股能力仍有限
- **推荐**：保留截面百分位（风险调整收益提升显著）

**截面百分位实现代码**：
```python
def _calculate_cross_sectional_percentile(self, df, features):
    """截面百分位：每天对所有股票进行排名
    
    与相对标签逻辑一致：判断"这只股票今天在所有股票中排第几"
    """
    for feat in features:
        if feat in df.columns:
            # 按日期分组，计算当日排名百分位
            df[f'{feat}_CS_Pct'] = df.groupby(df.index)[feat].rank(pct=True)
    return df
```

- ATR：ATR, ATR_Ratio
- 成交量：Volume_Ratio_5d, Volume_Ratio_20d
- 动量：Momentum_20d
- RSI：RSI

**代码位置**：`ml_services/ml_trading_model.py:4292-4328`

**教训**：
- **个股模型的选股能力有限**，不能单独依赖预测概率排序
- 预测分散度 > 0.1 表示模型有区分度，避免了"全涨全跌"问题
- IC 应作为选股策略的参考指标，而非唯一判断标准
- 个股预测需结合恒指择时判断，而非单独使用
- **滚动百分位不适合时间序列预测，更适合截面选股因子**
- **截面百分位与相对标签逻辑一致**：都是"横向比较"
- **单调约束 + 时间衰减是推荐配置**

### 3. 标签类型必须与模型目标匹配 ⭐（新增 2026-05-02）

**问题**：标签类型与模型目标不匹配，导致模型"答非所问"

**核心原则**：

| 问题类型 | 标签类型 | 模型学习目标 | 适用场景 |
|----------|----------|--------------|----------|
| **截面问题** | 相对标签 | 学 Alpha（跑赢其他股票） | 选股 |
| **时序问题** | 绝对标签 | 学 Beta/方向（涨还是跌） | 择时 |

**标签定义**：
- **相对标签**：`Return > Median_Return`（跑赢/跑输其他股票）
- **绝对标签**：`Return > 0`（涨/跌）

**模型目标与标签匹配**：

| 模型类型 | 目标 | 标签类型 | 原因 |
|----------|------|----------|------|
| **个股模型** | 选股 | 相对标签 | 目标是挑出跑赢的股票，必须比较 |
| **恒指模型** | 择时 | 绝对标签 | 目标是预测大盘方向，无需比较 |

**混用后果**：

| 错误做法 | 后果 |
|----------|------|
| 个股模型用绝对标签 | 模型变成"宏观择时器"，所有股票预测方向一致，无法选股 |
| 恒指模型用相对标签 | 无意义，恒指是单一标的，没有比较对象 |

**教训**：
- 标签定义决定模型学什么，混用等于让模型"答非所问"
- 选股问题必须用相对标签，择时问题必须用绝对标签
- 模型目标与标签类型必须匹配，否则模型会学到错误的信号

### 4. 特征残差化提升准确率 ⭐（新增 2026-04-29，更新 2026-05-03）

**问题**：全局特征（美债利率）导致模型依赖宏观信号，而非个股特异性特征

**解决方案**：
1. **相对标签**：以当天所有股票收益率中位数为基准
2. **特征残差化**：对 35 个微观特征剔除宏观因子贡献
3. **残差替换原始特征**（2026-05-03 实施）：`keep_original=False`

**残差化策略演进**：

| 阶段 | 实现方式 | 特征名 | 问题 | 解决 |
|------|---------|--------|------|------|
| 初始 | `keep_original=True` | `Momentum_20d_Residual` | 原始特征仍可使用，模型可走捷径 | ❌ |
| 最终 | `keep_original=False` | `Momentum_20d` | 残差直接替换，强制使用净化后特征 | ✅ |

**破坏性变更说明**：
- **旧模型不兼容**：`keep_original=False` 后，旧模型 pkl 文件无法加载（特征名不一致）
- **必须重新训练**：所有模型需重新训练以生成新的特征集
- **优势**：消除模型使用原始宏观特征的

### 8. CatBoost 分类特征 NaN 处理 ⭐（新增 2026-04-27）

**问题**：`predict_proba` 函数缺少分类特征 NaN 处理，导致测试数据预测失败

**现象**：所有 Walk-forward Fold 验证失败，报错 `cat_features must be integer or string, real number values and NaN values should be converted to string`

**原因**：
- 训练时对分类特征进行了 `fillna('unknown')` 和 `LabelEncoder` 转换
- 但 `predict_proba` 函数缺少相同处理

**解决**：
```python
# 在 predict_proba 中添加分类特征处理
for col in self.categorical_encoders.keys():
    if col in test_df.columns:
        test_df[col] = test_df[col].fillna('unknown').astype(str)
        encoder = self.categorical_encoders[col]
        test_df[col] = test_df[col].apply(
            lambda x: encoder.transform([x])[0] if x in encoder.classes_ else -1
        )
```

**教训**：
- 训练和预测的预处理逻辑必须一致
- 分类特征的 NaN 值必须在进入 CatBoost Pool 之前处理
- 建议使用统一的预处理管道

**代码**：`ml_services/ml_trading_model.py:4937-4948`

### 12. 网络特征不适合放入个股模型 ⭐（新增 2026-04-30）

**问题**：将网络特征（中心性、社区归属等）放入个股预测模型，效果不佳

**深层原因**：

| 原因 | 说明 |
|------|------|
| **模型假设冲突** | 个股模型假设 IID，但网络特征本质是描述"不独立性"，在市场极端波动时会导致预测偏误 |
| **维度灾难** | 网络特征呈长尾分布，极端值（权重股/妖股）会导致过拟合 |
| **信噪比低** | 网络拓扑每天变化 <0.1%，每日更新带来的只是计算开销，而非预测增量 |

**数据泄漏的本质**：

> "这是操作规范问题，而非特征本质问题。只要严格使用 $T-1$ 日数据构建网络来预测 $T$ 日，就不存在泄漏。"

当前移除了网络特征，但可以通过延迟处理来解决。

**正确用法**：

| 用法 | 说明 |
|------|------|
| **风险对冲** | 波动率网络密度 > 阈值 → 市场进入"同涨同跌"模式 → 降低仓位 |
| **逻辑触发更新** | 监控邻接矩阵 Frobenius 范数变化率，超过阈值才重新计算 |
| **残差化处理** | 网络特征对动量/成交量因子回归，只取残差作为输入 |

**教训**：
- 网络特征描述的是"全局结构"，与个股模型"局部预测"的目标不匹配
- 网络分析应作为独立的风险监控工具，而非预测特征
- 波动率网络密度是衡量市场"拥挤度"的绝佳指标

**代码**：`data_services/network_features.py`、`ml_services/stock_network_analysis.py`

## 四、开发规范

### CatBoost 参数调优（特征数量变化时）

**场景**：特征数量从 1045 减少到 730，需要调整参数

**调整原则**：

| 参数 | 特征减少时调整方向 | 原因 |
|------|-------------------|------|
| n_estimators | 增加 ↑ | 特征减少需要更多树来捕捉信息 |
| depth | 增加 ↑ | 允许更复杂的决策边界 |
| l2_leaf_reg | 降低 ↓ | 特征减少后过拟合风险降低 |
| subsample | 增加 ↑ | 特征减少可以增加行采样 |
| colsample_bylevel | 增加 ↑ | 特征减少可以增加列采样 |

**当前 20天模型参数（730 特征，2026-04-29 手动调优最优版）**：

| 参数 | 值 | 说明 |
|------|-----|------|
| n_estimators | 600 | 树数量（增加） |
| depth | 7 | 树深度（增加） |
| learning_rate | 0.03 | 学习率（降低） |
| l2_leaf_reg | 2 | L2 正则化（降低） |
| subsample | 0.75 | 行采样（增加） |
| colsample_bylevel | 0.75 | 列采样（增加） |

**调优效果对比（2026-04-29）**：

| 指标 | 旧参数 | 手动调优 | 系统调优 | 目标 | 最佳 |
|------|--------|---------|---------|------|------|
| 准确率 | 57.15% | 54.93% | 55.78% | >58% | 旧参数 |
| 夏普比率 | 0.8122 | **0.9059** | 0.8802 | >0.9 | 手动调优 ✅ |
| 最大回撤 | -0.53% | **-0.20%** | -0.22% | - | 手动调优 ✅ |
| 稳定性 | 0.5844 | 0.4770 | **0.3962** | - | 系统调优 |

**最终选择**：手动调优参数（夏普比率达标，最大回撤最优）

**教训**：
- 特征减少后应降低正则化强度
- 特征数量变化后必须重新验证性能
- 使用 Walk-forward 验证而非训练 CV 评估真实性能
- **降低学习率 + 增加树数量** 可提升夏普比率和稳定性
- **增加树深度** 可捕捉更复杂模式，但可能略微降低准确率
- **系统调优需谨慎**：快速验证结果可能过于乐观，需完整验证确认

### 调优脚本改进经验（2026-04-29）

**问题**：快速验证结果过于乐观，完整验证后泛化不足

## 七、分析器设计经验

### 近期指标权重配置

**场景**：风险回报率分析需更关注近期表现

**方案**：添加 `recent_bias` 参数动态调整权重

```bash
# 参数说明
--recent-bias 0.5  # 平衡近期和历史（默认）
--recent-bias 0.7  # 偏向近20日（推荐）
--recent-bias 1.0  # 完全使用近期数据
```

**实现要点**：
- 风险评分：混合近期和历史指标（最大回撤、波动率、Beta）
- 回报评分：提升近期表现权重（10%→17%），降低全周期指标（夏普比率 10%→5%）
- 混合公式：`score = recent * bias + historical * (1 - bias)`

**教训**：
- 不同时间窗口的指标有不同价值，应可配置
- 近期数据更反映当前市场状态，但历史数据提供稳定性
