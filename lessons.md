## 一、核心警告

### 1. 数据泄漏阈值因模型而异 ⭐（新增 2026-04-28）

**问题**：恒指和个股的数据泄漏阈值不同

**原因**：
- 恒指是"均值"，噪声被对冲，预测更容易
- 个股受公司特定事件影响，预测难度更高
- 因此恒指的"正常准确率上限"高于个股

**阈值标准**：

| 模型类型 | Walk-forward 准确率正常范围 | 数据泄漏信号 |
|---------|---------------------------|-------------|
| 个股 | 50-60% | >65% |
| 恒指 | 60-82% | >80%（增强模型） |

**教训**：
- 不要用同一标准衡量不同类型的模型
- 恒指 80% 准确率是正常的，个股 65% 就要警惕

### 2. 特征冗余清理需谨慎 ⭐（新增 2026-04-28）

**问题**：清理冗余特征后夏普比率下降 15-20%

**验证数据**：

| 指标 | 清理前 (1045) | 清理后 (730) | 变化 |
|------|--------------|--------------|------|
| 准确率 | 54.82% | 57.15% | +2.33% ✅ |
| 夏普比率 | 1.0221 | 0.8122 | -20.5% ⚠️ |
| 稳定性（夏普标准差） | 0.7597 | 0.5844 | +23% ✅ |

**教训**：
- **r=1.0 的"冗余"特征可能有预测价值**：在不同市场环境下可能有差异化信号
- CatBoost 对特征冗余有天然容错机制，正则化可自动处理
- 交叉特征减少会丢失非线性交互信息
- 清理特征后需重新调参，不能直接使用旧参数

**建议**：
- 清理前必须进行对比验证
- 保留性能最好的版本
- 特征工程变更后应调整模型参数

### 3. 恒指与个股的预测准确率差距大 ⭐（更新 2026-04-28）

**问题**：恒指预测准确率显著高于个股，个股预测接近随机

**验证数据**：

| 指标 | 恒指 | 个股平均 | 差异 |
|------|------|---------|------|
| 1天预测正确时 → 20天也正确 | **83.56%** | **58.20%** | +25.36% |
| 5天预测正确时 → 20天也正确 | **81.95%** | **61.86%** | +20.09% |
| 1天+5天都正确时 → 20天也正确 | **81.49%** | **61.74%** | +19.75% |
| 假突破(101)准确率 | **95.00%** | **54.58%** | +40.42% |
| 5d预测概率与20d实际方向相关性 | +0.35 | +0.03 | 个股相关性弱 |

**教训**：
- 恒指是"均值"，个股噪声被对冲，预测更容易
- 个股所有模式准确率接近随机水平（50%~62%）
- **不能将恒指策略直接套用于个股**
- 个股预测信号不可靠，需结合恒指确认趋势

**代码**：`ml_services/analyze_three_horizon_relationships.py`、`ml_services/analyze_stock_causal_chain.py`

### 4. 训练窗口不是越大越好 ⭐

**问题**：样本/特征比 7.6x 低于推荐 10x，但增加窗口反而降低准确率

**验证**：12个月 60.11% > 18个月 56.57% > 24个月 54.85%

**教训**：
- 港股近年结构性变化，近期数据比更多历史数据更有预测价值
- Walk-forward 机制保证时效性，弥补样本量不足
- CatBoost 正则化能有效处理高维问题

### 5. 1d 预测有"反向指标"效应 ⭐

**问题**：5个1d预测全错时，5d准确率高达 87.5%（高于全对的 67.4%）

**教训**：
- 1d预测噪声大，全错可能是"反向信号"
- 短期预测的大幅偏离可作为中长期趋势的辅助判断
- 但样本量较小（16次），需谨慎参考

### 6. 性能监控提前评估 Bug ⭐

**问题**：`fetch_price` 容错机制导致提前评估未来到期的预测

**教训**：
- "容错"机制可能引入隐蔽 Bug
- 时间相关逻辑必须有边界检查

**代码**：`ml_services/performance_monitor.py:179-182`

### 7. 特征缓存版本控制 ⭐

**问题**：新增特征后，旧缓存缺少新特征列，导致预测失败或使用错误特征

**解决**：
- 缓存验证时检查必需特征列是否存在
- 缺少新特征时标记缓存无效，重新计算

**教训**：
- 特征工程迭代时，缓存验证逻辑必须同步更新
- 检测特征列而非仅检测缓存文件存在性

**代码**：`ml_services/ml_trading_model.py`（CatBoost 缓存验证逻辑）

### 8. CatBoost 分类特征 NaN 处理 ⭐（新增 2026-04-27）

**问题**：`predict_proba` 函数缺少分类特征 NaN 处理，导致测试数据预测失败

**现象**：所有 Walk-forward Fold 验证失败，报错 `cat_features must be integer or string, real number values and NaN values should be converted to string`

**原因**：
- 训练时对分类特征进行了 `fillna('unknown')` 和 `LabelEncoder` 转换
- 但 `predict_proba` 函数缺少相同处理

**解决**：
```python
# 在 predict_proba 中添加分类特征处理
for col in self.categorical_encoders.keys():
    if col in test_df.columns:
        test_df[col] = test_df[col].fillna('unknown').astype(str)
        encoder = self.categorical_encoders[col]
        test_df[col] = test_df[col].apply(
            lambda x: encoder.transform([x])[0] if x in encoder.classes_ else -1
        )
```

**教训**：
- 训练和预测的预处理逻辑必须一致
- 分类特征的 NaN 值必须在进入 CatBoost Pool 之前处理
- 建议使用统一的预处理管道

**代码**：`ml_services/ml_trading_model.py:4937-4948`

### 12. 网络特征不适合放入个股模型 ⭐（新增 2026-04-30）

**问题**：将网络特征（中心性、社区归属等）放入个股预测模型，效果不佳

**深层原因**：

| 原因 | 说明 |
|------|------|
| **模型假设冲突** | 个股模型假设 IID，但网络特征本质是描述"不独立性"，在市场极端波动时会导致预测偏误 |
| **维度灾难** | 网络特征呈长尾分布，极端值（权重股/妖股）会导致过拟合 |
| **信噪比低** | 网络拓扑每天变化 <0.1%，每日更新带来的只是计算开销，而非预测增量 |

**数据泄漏的本质**：

> "这是操作规范问题，而非特征本质问题。只要严格使用 $T-1$ 日数据构建网络来预测 $T$ 日，就不存在泄漏。"

当前移除了网络特征，但可以通过延迟处理来解决。

**正确用法**：

| 用法 | 说明 |
|------|------|
| **风险对冲** | 波动率网络密度 > 阈值 → 市场进入"同涨同跌"模式 → 降低仓位 |
| **逻辑触发更新** | 监控邻接矩阵 Frobenius 范数变化率，超过阈值才重新计算 |
| **残差化处理** | 网络特征对动量/成交量因子回归，只取残差作为输入 |

**教训**：
- 网络特征描述的是"全局结构"，与个股模型"局部预测"的目标不匹配
- 网络分析应作为独立的风险监控工具，而非预测特征
- 波动率网络密度是衡量市场"拥挤度"的绝佳指标

**代码**：`data_services/network_features.py`、`ml_services/stock_network_analysis.py`

## 四、开发规范

### CatBoost 参数调优（特征数量变化时）

**场景**：特征数量从 1045 减少到 730，需要调整参数

**调整原则**：

| 参数 | 特征减少时调整方向 | 原因 |
|------|-------------------|------|
| n_estimators | 增加 ↑ | 特征减少需要更多树来捕捉信息 |
| depth | 增加 ↑ | 允许更复杂的决策边界 |
| l2_leaf_reg | 降低 ↓ | 特征减少后过拟合风险降低 |
| subsample | 增加 ↑ | 特征减少可以增加行采样 |
| colsample_bylevel | 增加 ↑ | 特征减少可以增加列采样 |

**当前 20天模型参数（730 特征，2026-04-29 手动调优最优版）**：

| 参数 | 值 | 说明 |
|------|-----|------|
| n_estimators | 600 | 树数量（增加） |
| depth | 7 | 树深度（增加） |
| learning_rate | 0.03 | 学习率（降低） |
| l2_leaf_reg | 2 | L2 正则化（降低） |
| subsample | 0.75 | 行采样（增加） |
| colsample_bylevel | 0.75 | 列采样（增加） |

**调优效果对比（2026-04-29）**：

| 指标 | 旧参数 | 手动调优 | 系统调优 | 目标 | 最佳 |
|------|--------|---------|---------|------|------|
| 准确率 | 57.15% | 54.93% | 55.78% | >58% | 旧参数 |
| 夏普比率 | 0.8122 | **0.9059** | 0.8802 | >0.9 | 手动调优 ✅ |
| 最大回撤 | -0.53% | **-0.20%** | -0.22% | - | 手动调优 ✅ |
| 稳定性 | 0.5844 | 0.4770 | **0.3962** | - | 系统调优 |

**最终选择**：手动调优参数（夏普比率达标，最大回撤最优）

**教训**：
- 特征减少后应降低正则化强度
- 特征数量变化后必须重新验证性能
- 使用 Walk-forward 验证而非训练 CV 评估真实性能
- **降低学习率 + 增加树数量** 可提升夏普比率和稳定性
- **增加树深度** 可捕捉更复杂模式，但可能略微降低准确率
- **系统调优需谨慎**：快速验证结果可能过于乐观，需完整验证确认

### 调优脚本改进经验（2026-04-29）

**问题**：快速验证结果过于乐观，完整验证后泛化不足

**改进措施**：

| 改进项 | 旧版 | 新版 | 原因 |
|--------|------|------|------|
| 股票数量 | 10只 | 20只（快速）/59只（完整） | 更多样本减少过拟合 |
| 验证fold数 | 3个 | 4个（快速）/8个（完整） | 更全面评估 |
| 验证方法 | TimeSeriesSplit | Walk-forward风格 | 更接近真实预测 |
| 稳定性指标 | 仅准确率std | 准确率std + 夏普std | 全面评估稳定性 |
| 早停机制 | 无 | 连续5次无改进 | 避免无效搜索 |

**使用建议**：
```bash
# 标准调优（推荐）
python3 ml_services/hyperparameter_tuner.py --horizon 20 --n-iter 30

# 快速测试
python3 ml_services/hyperparameter_tuner.py --horizon 20 --n-iter 15 --quick

# 完整调优（耗时较长但最可靠）
python3 ml_services/hyperparameter_tuner.py --horizon 20 --n-iter 50 --full
```

**教训**：
- 调优脚本评估结果需用完整 Walk-forward 验证确认
- 快速模式适合初步探索，完整模式用于最终确认
- 早停机制可节省时间，但需设置合理阈值

## 七、分析器设计经验

### 近期指标权重配置

**场景**：风险回报率分析需更关注近期表现

**方案**：添加 `recent_bias` 参数动态调整权重

```bash
# 参数说明
--recent-bias 0.5  # 平衡近期和历史（默认）
--recent-bias 0.7  # 偏向近20日（推荐）
--recent-bias 1.0  # 完全使用近期数据
```

**实现要点**：
- 风险评分：混合近期和历史指标（最大回撤、波动率、Beta）
- 回报评分：提升近期表现权重（10%→17%），降低全周期指标（夏普比率 10%→5%）
- 混合公式：`score = recent * bias + historical * (1 - bias)`

**教训**：
- 不同时间窗口的指标有不同价值，应可配置
- 近期数据更反映当前市场状态，但历史数据提供稳定性
